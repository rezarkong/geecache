# 面试问答稿

## 1. 这个项目是做什么的？

这是一个用 Go 实现的分布式缓存原型。核心目标是把本地缓存、远端节点拉取和并发场景下的重复回源控制串起来。项目里实现了 LRU、本地缓存、一致性哈希、`singleflight`、gRPC + Protobuf 节点通信，并在后续补充了 TTL、空值缓存、peer 超时重试、熔断和运行时指标。

## 2. 为什么要用 `singleflight`？

因为热点 key 在高并发下容易造成重复回源，也就是缓存击穿。`singleflight` 可以保证同一个 key 在同一时刻只会有一个真实加载请求在执行，其他 goroutine 等待这次结果即可。这样可以把热点 key 的回源次数从 N 次压缩到 1 次。

## 3. 为什么要用一致性哈希？

因为分布式缓存需要决定某个 key 应该由哪个节点负责。如果简单取模，节点增减时大量 key 会重映射；一致性哈希可以让节点变化时只影响局部 key。项目里还用了虚拟节点，降低 key 分布不均的问题。

## 4. 本地缓存和远端 peer 是什么关系？

本地缓存是每个节点自己的热点数据副本，远端 peer 是一致性哈希选中的 owner。读流程是先查本地，miss 后优先找 owner 节点拉取；如果远端节点临时失败，再走本地 getter 降级。这样可以兼顾访问性能和节点间的数据分布。

## 5. 为什么要区分 peer 失败和 peer `not found`？

这是后面专门补的一个点。  
如果 peer 返回的是超时、网络错误或 5xx，这说明 owner 可能临时不可用，可以选择重试或降级回本地。  
但如果一致性哈希选中的 owner 明确返回 `not found`，说明这个 key 在 owner 上就是不存在的，此时再回本地重复查一次通常没有意义，容易造成错误语义和无效回源，所以我把这两类情况分开处理了。

## 6. 为什么要加空值缓存？

这是为了解决缓存穿透。对于不存在的 key，如果每次请求都直接打到后端数据源，就会造成重复 miss。项目里对 `ErrNotFound` 支持短 TTL 空值缓存，让同一个 missing key 在 TTL 窗口内直接返回，减少无效回源。

## 7. 为什么正常缓存值也要加 TTL 和抖动？

只做 LRU 不够，因为缓存项可能长期不更新，也可能在同一时间集中失效。TTL 可以控制数据的新鲜度，抖动可以把本来同时过期的一批 key 打散，降低缓存雪崩风险。

## 8. peer 重试为什么还要配熔断？

单纯重试只能解决短暂抖动，但如果某个 peer 持续超时或异常，继续打它只会拖慢整体读链路。  
所以项目里在重试之外补了简单熔断：连续失败达到阈值后短时间内不再请求这个节点，直接走本地 fallback。这样可以减少无效等待。

## 9. 为什么要用 `context.Context`？

这是为了把超时和取消信号贯穿整个调用链。项目后面把 `Getter` 和 `PeerGetter` 都升级成了带 `context` 的接口，gRPC peer 请求也会直接复用调用方上下文。这样上游超时、取消、trace 信息都可以继续往下传。

## 10. 这个项目的指标主要看什么？

我重点看几类：

- `requests`、`cache_hits`、`cache_misses`、`hit_rate`
- `peer_attempts`、`peer_loads`、`peer_load_failures`、`peer_not_founds`
- `peer_retries`、`peer_fallbacks`、`circuit_open_events`、`circuit_rejects`
- `avg_peer_attempt_nanos`、`avg_peer_load_nanos`、`avg_peer_failure_nanos`
- `avg_local_load_nanos`

这些指标可以帮助判断热点 key 合并效果、远端节点稳定性、降级频率和本地/远端加载耗时差异。

## 11. 这个项目里最关键的测试有哪些？

- 并发 singleflight 测试，验证同 key 并发只回源一次
- gRPC 集成测试，验证节点间通过 gRPC + Protobuf 拉取
- peer 失败回退测试，验证远端失败时能走本地 getter
- peer `not found` 测试，验证不会错误回退到本地
- TTL 与空值缓存测试，验证正常过期和 missing key 场景
- `go test -race`，验证新增并发路径没有明显数据竞争

## 12. 如果继续优化，你还会做什么？

我会优先做三件事：

1. 引入真实的服务发现或配置热更新，而不是 demo 级的静态节点列表
2. 把指标接到 Prometheus/OpenTelemetry，而不是只返回 JSON
3. 用新版 `protoc` 工具链重新生成 protobuf 文件，避免手工维护生成代码

## 13. 如果面试官问“这个项目最大的收获是什么”？

可以这么回答：

这个项目最大的收获不是把 LRU 或一致性哈希写出来，而是把一个教程式缓存原型逐步补成一个更接近真实服务的实现。包括并发去重、异常节点处理、缓存穿透治理、指标设计、语义边界和测试验证，这些比单纯把功能跑通更接近实际工程问题。
